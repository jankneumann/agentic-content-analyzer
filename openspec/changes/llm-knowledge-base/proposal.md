# Proposal: LLM Knowledge Base

**Change ID**: `llm-knowledge-base`
**Status**: Proposed
**Created**: 2026-04-04

## Why

The newsletter aggregator currently treats **content items** as first-class citizens — each piece of ingested content gets summarized, theme-analyzed, and folded into digests. But the system lacks a persistent, evolving understanding of **topics and concepts** across content. Themes are extracted per-analysis as ephemeral `ThemeData` Pydantic models stored as JSON blobs in `ThemeAnalysis` records, then discarded after digest creation.

This means:
- **No topic memory** — the system re-discovers "RAG Architecture" from scratch each time rather than building on what it already knows
- **No compiled knowledge** — there's no persistent article explaining what the system understands about a topic, how it has evolved, and what evidence supports it
- **No self-navigation** — agents must rely on hybrid search for every query rather than reading a structured index of accumulated knowledge
- **No knowledge health** — there's no way to detect stale, contradictory, or incomplete understanding

Inspired by [Andrei Karpathy's approach](https://x.com/karpathy/status/1908190230636302686) to LLM-compiled personal knowledge bases, this feature promotes topics to first-class persistent entities with LLM-compiled markdown articles, auto-maintained indices, Q&A capabilities, and health checks. The database is the source of truth; Obsidian serves as an export/interface layer for visualization and feedback.

### Alignment with Existing Work

- **Theme analysis** (`theme-analysis` spec) already extracts themes — this feature persists them as `Topic` entities
- **Agentic analysis** (`agentic-analysis-agent` change) defines specialist agents — this adds a new KB specialist and gives KB tools to the Research specialist
- **Document search** (`document-search` spec) provides hybrid BM25+vector search — topics become searchable through existing infrastructure
- **Graphiti abstraction** (in-progress FalkorDB migration) — all graph access goes through `GraphitiClient`, supporting both Neo4j and FalkorDB backends

## What Changes

### New Components

1. **Topic Model** (`src/models/topic.py`) — Persistent SQLAlchemy entity promoting `ThemeData` to a first-class DB record. Full superset of ThemeData fields: name, slug, category (FK → TopicCategory), trend, relevance/novelty/strategic/tactical scores, related topics, parent-child hierarchy, and LLM-compiled article markdown. Lifecycle: DRAFT → ACTIVE → STALE → ARCHIVED (or MERGED).

2. **TopicCategory Model** (`src/models/topic_category.py`) — Dedicated taxonomy table replacing the fixed `ThemeCategory` enum for topic categorization. Hierarchical: each category has an optional `parent_id` (self-referential FK) enabling tree structures like `ML/AI → LLMs → Fine-tuning`. Fields: id, slug, name, description, parent_id, icon, color, display_order. Seeded with the 8 existing ThemeCategory values as top-level categories. New categories can be added via API/CLI without migrations. Maps directly to Obsidian folder structure.

3. **TopicNote Model** (`src/models/topic_note.py`) — Annotations on topics from users, agents, or the system. Supports filing Q&A answers, corrections, and observations back into the KB. Each note tracks whether it has been incorporated into the topic article.

3. **KB Compilation Service** (`src/services/kb_compiler.py`) — Incremental compiler that:
   - Gathers new evidence (recent ThemeAnalysis results, new content, agent insights)
   - Matches evidence to existing topics (exact name → update; semantic similarity → merge candidate; no match → create DRAFT)
   - LLM generates/updates article markdown per topic
   - Updates inter-topic relationships via GraphitiClient
   - Regenerates cached index files
   - Runs automatically after theme analysis in the daily pipeline AND on-demand via CLI/API

4. **KB Index Generator** (`src/services/kb_index_generator.py`) — Generates auto-maintained index files stored as special Topic records (slug prefixed with `_`):
   - `_index`: Master index — all topics with 1-line summaries
   - `_by_category`: Topics grouped by ThemeCategory
   - `_by_trend`: Topics grouped by ThemeTrend (emerging, growing, etc.)
   - `_relationship_map`: Topic adjacency list for navigation
   - `_coverage_gaps`: Categories with few/stale topics (generated by health check)

5. **KB Q&A Agent** — New KnowledgeBase specialist agent that answers questions against the compiled KB (not raw content). Answers can be filed back as TopicNotes. KB tools (`get_topic`, `search_kb`, `get_kb_index`) are also exposed to the Research specialist for cross-cutting queries.

6. **KB Health Check Service** (`src/services/kb_health.py`) — Periodic LLM sweeps:
   - Stale detection (no new evidence in >30 days)
   - Merge candidate detection (high semantic similarity between topics)
   - Coverage gap analysis (sparse categories)
   - Consistency checks (contradictory claims across topic articles)
   - Article quality scoring

7. **CLI Commands** (`src/cli/kb_commands.py`) — `aca kb compile|list|show|index|query|lint` command group

8. **API Endpoints** (`src/api/kb_routes.py`) — RESTful CRUD for topics, notes, compilation triggers, index retrieval, health checks, and Q&A

9. **MCP Tools** — `search_knowledge_base`, `get_topic`, `update_topic`, `add_topic_note`, `get_kb_index`, `compile_knowledge_base`

10. **Obsidian Export Service** (`src/services/kb_obsidian_export.py`) — Generates an Obsidian-compatible vault from the knowledge base:
    - **Folder structure**: TopicCategory hierarchy maps directly to folders (e.g., `ML-AI/LLMs/Fine-tuning/`)
    - **Topic files**: One `.md` file per topic with YAML frontmatter (slug, category path, trend, scores, dates) and article body
    - **Wikilinks**: `related_topic_ids` rendered as `[[Topic Name]]` wikilinks for Obsidian graph view
    - **Index files**: Cached indices exported as `_index.md` files in the vault root
    - **Images**: Referenced via relative paths (future: download to vault)
    - **CLI**: `aca kb export --format obsidian --output ./vault`
    - **API**: `GET /api/v1/kb/export/obsidian` returns a ZIP archive

### Modified Components

10. **Pipeline Runner** — Add KB compilation step after theme analysis in `aca pipeline daily`

11. **Theme Analyzer** — After extracting themes, call KB compiler to persist/update Topic entities

12. **Models `__init__.py`** — Export new Topic, TopicNote, TopicStatus models

13. **Settings** — Add `settings/knowledge_base.yaml` for compilation config (staleness threshold, max topics per compile, index refresh strategy) and prompt templates in `settings/prompts.yaml`

## Approaches Considered

### Approach 1: Database-First with Obsidian Export (Recommended)

Topics are persistent SQLAlchemy models in PostgreSQL. The database is the single source of truth. Obsidian vault is a derived export: `.md` files with YAML frontmatter generated from DB records, wikilinks from relationship data, and index files from cached indices. Edits in Obsidian are synced back via MCP tools or CLI import.

**Pros:**
- Leverages existing hybrid search (BM25 + pgvector) for topic discovery
- Full SQL queryability (filter by category, trend, date range, scores)
- Typed relationships with confidence scores via GraphitiClient (backend-agnostic)
- Natural integration with existing pipeline, agents, API, and MCP infrastructure
- Multi-user capable with permissions and concurrent access
- Versioning via `article_version` column with full revision history
- Scales to 100K+ topics without degradation

**Cons:**
- Requires Alembic migration and new model maintenance
- Obsidian export is a derived view, not the canonical representation — adds sync complexity
- More infrastructure than flat files for small-scale personal use

**Effort:** L (large — new model, service, CLI, API, MCP, pipeline integration, agent)

### Approach 2: Flat Markdown Files (Karpathy's Original)

Topics are `.md` files in a directory structure, navigated directly by LLMs via file read/write tools. Obsidian IS the representation. No database involvement for KB content.

**Pros:**
- Zero schema — just files; extremely simple to start
- Git-native version control with human-readable diffs
- LLMs can directly read/write files without API abstraction
- Perfect Obsidian integration — files ARE the vault

**Cons:**
- No semantic search — relies on LLM reading index files or naive grep
- No typed relationships — only wikilinks (`[[Topic]]`), no confidence scores or relationship types
- Degrades past ~1000 topics — file enumeration and full-text search become expensive
- Disconnected from existing pipeline (digests, agents, search) — requires separate integration
- Single-user only; no concurrent access or permissions
- Cannot answer structured queries ("find all EMERGING topics in ML_AI") without reading every file

**Effort:** M (medium — simpler model but large integration effort to connect with existing infra)

### Approach 3: Hybrid — Lightweight DB + Filesystem Sync

Topics have a lightweight DB record (name, slug, category, scores) but the article markdown lives as `.md` files in a configurable directory. A bidirectional sync layer keeps DB metadata and file content in sync. Obsidian points at the file directory.

**Pros:**
- DB handles structured queries and search
- Files are directly editable in Obsidian without export/import
- Git-friendly article content

**Cons:**
- Bidirectional sync is notoriously complex (conflict resolution, race conditions)
- Two sources of truth — debugging discrepancies is painful
- File system as storage limits deployment options (Railway, serverless)
- Still need the full DB model for metadata, just with the complication of split storage

**Effort:** L (large — same DB work as Approach 1 plus file sync complexity)

### Selected Approach

**Approach 1: Database-First with Obsidian Export** — selected because it builds naturally on existing infrastructure (SQLAlchemy, hybrid search, GraphitiClient, pipeline). The graph backend abstraction (Graphiti supporting both Neo4j and FalkorDB) means topic relationships are stored portably. Obsidian export is a one-way derivation that avoids the bidirectional sync pitfalls of Approach 3.

Key refinements from discovery:
- **KB Q&A**: New KnowledgeBase specialist agent, with KB tools also available to the Research specialist
- **Topic model**: Full superset of ThemeData fields — self-contained, no joins needed
- **Index storage**: Cached markdown as special Topic records (slug `_index`, `_by_category`, etc.)
- **Compilation trigger**: Automatic in daily pipeline (after theme analysis) + on-demand via CLI/API

## Scope

### In Scope
- Topic + TopicCategory + TopicNote SQLAlchemy models + Alembic migration
- TopicCategory hierarchical taxonomy (replaces fixed ThemeCategory enum for topics)
- KB compilation service (incremental)
- KB index generator (cached markdown indices)
- KB Q&A agent (new specialist + tools on Research specialist)
- KB health check / linting service
- Obsidian export specification and implementation (folder structure, frontmatter, wikilinks)
- CLI commands (`aca kb`)
- API endpoints (`/api/v1/kb/`)
- MCP tools (6 tools)
- Pipeline integration (compile step after theme analysis)
- Settings YAML + prompt templates

### Out of Scope (deferred)
- Obsidian import / sync-back from edited vault files (future)
- Voice mode integration / audio briefs (Phase 3)
- Interactive HTML output generation (Phase 3)
- Synthetic data generation / fine-tuning from KB (future)
- Frontend UI for KB browsing (separate change)

## Risks

| Risk | Mitigation |
|------|------------|
| Topic deduplication is hard (LLM may generate "RAG" and "Retrieval-Augmented Generation" as separate topics) | Semantic similarity check during compilation; merge candidate detection in health checks |
| Compilation adds 30-60s to daily pipeline | Incremental compilation (only new evidence); configurable skip via `--skip-kb` flag |
| Graph backend abstraction (FalkorDB migration) may change relationship storage patterns | All graph access through GraphitiClient; no direct database queries |
| Topic article quality may vary across LLM providers | Evaluation framework (LLM-as-Judge) already exists; extend to KB articles |
| Large topic count may slow index regeneration | Incremental index updates; only regenerate indices for affected categories |
