## 2025-05-27 - Information Leakage via Exception Details
**Vulnerability:** The `upload_document` endpoint in `src/api/upload_routes.py` was catching `ValueError` and returning `str(e)` to the client. This exposed potential secrets or internal paths if underlying libraries raised `ValueError` with sensitive details.
**Learning:** Using standard exceptions like `ValueError` for both expected validation errors (safe to expose) and unexpected internal errors (unsafe) makes it hard to handle them securely at the API layer.
**Prevention:** Define custom exceptions (e.g., `FileIngestionError`) for expected, safe-to-expose errors. Catch these specific exceptions in API routes. Allow unexpected exceptions to bubble up to a global error handler that returns generic 500 responses and logs details server-side.

## 2025-05-28 - Information Leakage via Global Database Exception Details
**Vulnerability:** The global FastAPI exception handlers for `DataError` and `AsyncpgDataError` in `src/api/middleware/error_handler.py` were returning the raw database error strings (`exc.orig` or `exc`) directly in the 422 Unprocessable Entity HTTP response. This exposed internal PostgreSQL-level schema, type, and encoding details to any client providing invalid parameters.
**Learning:** Returning `str(exc)` from any database-level driver exception directly to an API consumer is an information leakage risk. Even if the error is due to user input validation, the driver's default error string may contain unexpected internals.
**Prevention:** Global exception handlers that catch low-level library or driver errors (like SQLAlchemy or asyncpg errors) must sanitize the response payload by substituting raw exception messages with generic strings (e.g., "Invalid parameter value provided."), while ensuring the actual exception is still logged securely server-side for debugging.
